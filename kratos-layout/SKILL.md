# Kratos Project Layout

## Overview

This skill provides a guide to the standard project layout used in Kratos applications, which is generated by `kratos new`. This structure is designed to promote clean architecture, separation of concerns, and maintainability by organizing code into distinct layers with clear responsibilities.

## When to Use

Use this skill when you need to:
- Understand where to place new code (e.g., business logic, data access, API services).
- Follow the flow of a request through the different layers of the application.
- Know the purpose of each directory generated by the Kratos CLI.

## Directory Structure and Dependency Flow

A Kratos project is organized into several key directories. The dependency rule is that outer layers can depend on inner layers, but not the other way around.

**Flow: `server` → `service` → `biz` → `data`**

```
. (Project Root)
├── api/          # (Contract) Protobuf API definitions and generated Go code.
|
├── cmd/          # (Entrypoint) Application entry point, wires everything together.
|   └── server/
|
├── configs/      # (Configuration) Local config files (e.g., config.yaml).
|
└── internal/     # (Private Logic) All private application code.
    |
    ├── server/   # (Delivery) HTTP & gRPC server instances. Depends on `service`.
    |
    ├── service/  # (Service) Implements the Protobuf service interface. Depends on `biz`.
    |
    ├── biz/      # (Business) Business logic and use cases. Depends on `data` interfaces.
    |
    └── data/     # (Data Access) Repository implementations, DB connections.
```

### Layer Responsibilities

- **`api/`**: The public contract of your service. This is the only package that other services should import.
- **`cmd/server/`**: The application's `main` package. Its only job is to initialize and run the application by calling into `internal` and using Wire for dependency injection.
- **`internal/server/`**: Creates and configures the HTTP and gRPC server instances. It receives the `service` layer as a dependency.
- **`internal/service/`**: Implements the gRPC service interface defined in your `.proto` files. It acts as a translator between the transport layer (HTTP/gRPC) and the business logic. It should contain minimal logic and primarily call methods on the `biz` layer.
- **`internal/biz/`**: The core business logic layer. It contains use cases and domain objects. It defines interfaces for its data requirements (e.g., `GreeterRepo`) and depends on those interfaces, not on concrete data implementations.
- **`internal/data/`**: The data access layer. It implements the repository interfaces defined in the `biz` layer. It handles all database connections, ORM logic (like Ent), and external API calls.

## Common Mistakes

- **Putting Business Logic in the `service` Layer:** The `service` layer should only handle request/response mapping. All core business logic belongs in the `biz` layer. This keeps your business logic transport-agnostic.
- **`biz` Layer Depending on `data` Concrete Types:** The `biz` layer should define its own repository interfaces and depend on them. The `data` layer then implements these interfaces. This inverts the dependency and makes your business logic independent of the data storage implementation.
- **Exposing `internal` Packages:** The `internal` directory is a Go convention that prevents other projects from importing your private code. Never expose types from the `internal` package in your public `api`.
